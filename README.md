# Event Accommodation Coordinator

A web app that matches event guests to volunteer hosts for accommodation. Handles host reconfirmation, guest intake, matching, and two-sided confirmation with contact exchange.

## How it works

1. **Coordinator (you)** imports last year's hosts from Excel, then triggers reconfirmation — each host gets an email + SMS with a unique link to confirm availability for this year.
2. **Guests** fill a public form with their dates and group size.
3. **Coordinator** reviews proposed matches (generated by the matching engine) and approves them.
4. **Both parties** receive notifications with a confirm/decline link.
5. **Once both confirm**, contact details are automatically exchanged via email.

## Tech stack

- **Next.js 14** (App Router) — frontend + API routes
- **Supabase** — Postgres DB, auth, row-level security
- **Resend** — transactional email
- **Twilio** — SMS notifications
- **Tailwind CSS** — styling
- **Deployed on Vercel** — free tier

### 7. Trigger host reconfirmation

The app uses **sequential outreach** to contact hosts respectfully. The schedule and channel order are **configurable** via env vars — no code changes needed.

**Default schedule** (configurable):

| Step | Days | Channel | Stage label |
|------|------|---------|-------------|
| 0 | Day 0 | SMS + email together | `sent_initial` |
| 1 | Day 2 | SMS reminder | `sent_sms_2` |
| 2 | Day 4 | Email reminder | `sent_email_2` |
| 3 | Day 6 | Automated voice call (Twilio TTS) | `sent_voice` |
| Final | Day 8 | Flagged for manual call | `manual_required` |

Hosts who respond at any step drop out of the queue automatically. If a host has no phone, SMS/voice steps are skipped.

**Configuring the schedule** — in `.env.local`:

```dotenv
# Days between each stage (default: 2). Range: 1-30.
OUTREACH_STAGE_DELAY_DAYS=2

# Comma-separated channel sequence. Allowed: sms, email, sms+email, voice
OUTREACH_CHANNEL_SEQUENCE=sms+email,sms,email,voice
```

**Common alternative configurations:**

```dotenv
# Skip voice calls — gentler, no robocall reputation risk
OUTREACH_CHANNEL_SEQUENCE=sms+email,sms,email

# Email-first, slow pace (3 days between stages)
OUTREACH_STAGE_DELAY_DAYS=3
OUTREACH_CHANNEL_SEQUENCE=email,sms,email,voice

# Aggressive — 1 day between stages, voice call early
OUTREACH_STAGE_DELAY_DAYS=1
OUTREACH_CHANNEL_SEQUENCE=sms+email,voice,email,sms

# Email-only, three reminders
OUTREACH_CHANNEL_SEQUENCE=email,email,email
```

**After changing env vars:** stop and restart the dev server (`Ctrl+C` then `npm run dev`). On Vercel, redeploy or restart the function. The active configuration is shown in the coordinator dashboard's Outreach tab so you can verify what's running.

**Trigger options:**
- **Manual:** Coordinator dashboard → Outreach tab → "Run outreach now"
- **Automatic:** Vercel Cron runs `/api/outreach/run` daily at 14:00 UTC (in `vercel.json`)

The voice call uses Twilio Programmable Voice with text-to-speech. The host hears: *"Hi [Name], this is the accommodation team for [Event]. To confirm yes, press 1. To decline, press 2. To repeat, press 9."* Voicemail detection leaves a short message instead of the menu.

After all 5 attempts, hosts appear in the "Manual call required" list on the dashboard with their phone number as a clickable `tel:` link. Mark each as Yes/No/DNC after calling.

## Guest voice intake (inbound calls)

Guests can call your Twilio number to start their accommodation request by voice. The flow is hybrid — voice captures the noisy stuff (name + dates + party size), then sends an SMS with a link to a small page where the guest just enters their email and confirms.

**Configure Twilio for inbound:**

1. In the Twilio console, open your phone number
2. Under **Voice Configuration → A call comes in**:
   - Set to "Webhook"
   - URL: `{NEXT_PUBLIC_SITE_URL}/api/voice/inbound`
   - HTTP method: `POST`
3. Under **Call Status changes**:
   - URL: `{NEXT_PUBLIC_SITE_URL}/api/voice/inbound-status`
   - HTTP method: `POST`

**Test the flow:**

1. Call your Twilio number from a phone that can receive SMS
2. Speak your name when prompted (e.g. "Maria Lopez")
3. Press 1 to confirm
4. Enter arrival as 4 digits (e.g. `0815` for August 15)
5. Enter departure as 4 digits (e.g. `0817` for August 17)
6. Enter party size + `#` (e.g. `2#`)
7. Hang up — within seconds you'll get an SMS with a link
8. Tap the link → confirm details + add email → submit
9. Check the coordinator dashboard's "Intake" tab to see the session, and "Guests" tab to see the new guest row

**Why hybrid (voice + SMS)?**
Email addresses are notoriously difficult to capture by voice. Either spelling them out by keypad is brutal for guests, or speech-to-text mangles them. Sending an SMS link works around this by letting guests type their email on their own phone, where typos can be caught.

If the caller's phone can't receive SMS, the dashboard still shows the partial session in the Intake tab — you can call them back manually and complete the request through the regular `/guest` form.

## URLs

- `/` — landing page (3 cards: guest request, host signup, coordinator)
- `/guest` — public guest intake form (share with guests)
- `/host/signup` — public host signup form (share with potential hosts)
- `/host/[token]` — host reconfirmation (yes/no + capacity) — sent via email/SMS
- `/host/[token]/edit` — host self-service profile editor (full details)
- `/intake/[token]` — voice-call follow-up form for guests (sent via SMS after their call)
- `/coordinator` — admin dashboard (password-protected)

## Host signup approval workflow

When a new host fills `/host/signup`:

1. They get an immediate email: *"Thanks, a coordinator will review and confirm shortly."*
2. The coordinator (you) get an alert email at `COORDINATOR_EMAIL` with a link to the dashboard.
3. The host appears in the **Pending approvals** section at the top of the Hosts tab.
4. You click **Approve** or **Reject**:
   - **Approve** → host is confirmed available and gets a "you're in!" email with their profile management link
   - **Reject** → host gets a polite "thank you for offering, but..." email; you can optionally add a custom note

Only `approved` hosts are considered by the matching algorithm. Hosts imported from Excel default to `approved` automatically.

**Spam protection on the public form:**
- Hidden honeypot field (legit users won't fill it; bots will)
- Disposable email domain blocklist (mailinator, tempmail, etc.)
- Duplicate-email rejection (can't sign up twice with same email)
- 30-guest capacity cap

To configure the coordinator alert email, add `COORDINATOR_EMAIL=you@yourdomain.com` to your `.env.local`.

## Coordinator access

Set `COORDINATOR_PASSWORD` in env. Login at `/coordinator` with that password. (For production, consider upgrading to Supabase Auth with email login.)

## Matching logic

The matcher pairs each guest with a host where:
- Host has `confirmed_available = true`
- Host capacity >= guest group size
- Host is not already assigned to another unconfirmed guest for overlapping dates

You can override matches manually in the coordinator dashboard before sending notifications.

## Security notes

- Host confirmation tokens are UUIDs — unguessable, single-purpose per host.
- Contact details (phone, address) are never exposed to guests until BOTH parties confirm the match.
- The guest form is public but rate-limited at the API route.
- The coordinator route uses a password env var; for stronger auth, swap in Supabase Auth.

## Troubleshooting

- **Emails not arriving:** Check Resend dashboard → Logs. Most common issue is unverified sender domain.
- **SMS not arriving:** Check Twilio console → Monitor → Logs. Trial accounts can only text verified numbers.
- **Supabase errors:** Check you ran both `schema.sql` AND `policies.sql`.
